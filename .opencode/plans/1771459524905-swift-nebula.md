# Plan: Locked 30/60 FPS Recording Under GPU Strain

## Goal
Ensure consistent frame rate recording even when GPU is under heavy load by implementing non-blocking capture, backpressure signaling, and adaptive FPS.

## Current Bottlenecks
1. **Blocking capture send** (`dxgi/types.rs:460`) - `frame_tx.send()` blocks when encoder channel is full
2. **Blocking encoder write** (`hw_encoder/types.rs:837-839`) - `stdin.write_all()` stalls pipeline if FFmpeg slow
3. **No backpressure** - encoder overload doesn't signal capture to slow down
4. **No FPS adaptation** - tries to catch up instead of adapting frame rate

## Implementation

### Phase 1: Non-Blocking Capture with Frame Dropping
**File:** `src/capture/dxgi/types.rs:460`

Change blocking `send()` to non-blocking `try_send()`:
```rust
// Replace line 460
match frame_tx.try_send(frame) {
    Ok(()) => { /* existing success logic */ }
    Err(TrySendError::Full(_)) => {
        dropped_count += 1;
        if dropped_count % 60 == 0 {
            warn!("Dropped {} frames (encoder behind)", dropped_count);
        }
    }
    Err(TrySendError::Disconnected(_)) => break,
}
```

### Phase 2: Backpressure Signaling
**New file:** `src/capture/backpressure.rs`

Shared atomic state between capture and encoder:
- `queued_frames: AtomicU32` - frames in channel
- `max_queued_frames: AtomicU32` - threshold for dropping (default 8)
- `encoder_overloaded: AtomicBool` - encoder can't keep up
- `fps_divisor: AtomicU32` - 0=full, 1=half (30fps), 2=third (20fps)

Integration:
- Capture checks `should_drop_frame()` before sending
- Encoder updates after each encode
- Wire through `app.rs` to both threads

### Phase 3: Async Frame Writer
**New file:** `src/encode/frame_writer.rs`

Decouple `stdin.write_all()` from encoder thread:
- Dedicated writer thread with bounded queue (16 frames)
- Non-blocking `try_queue()` for frame submission
- Reports write latency to backpressure system

**Modify:** `src/encode/hw_encoder/types.rs:834-839`
- Replace direct `stdin.write_all()` with `async_writer.try_queue()`

### Phase 4: Dynamic FPS Adaptation
**File:** `src/capture/dxgi/types.rs`

Add `FpsAdaptation` state to `capture_loop()`:
- Track consecutive drops vs successes
- After 10 consecutive drops: reduce FPS (60→30→20)
- After 100 consecutive successes: restore FPS
- 3-second cooldown between adaptations

```rust
struct FpsAdaptation {
    current_divisor: u32,      // 0=60fps, 1=30fps, 2=20fps
    consecutive_drops: u32,
    consecutive_success: u32,
    last_adaptation: Instant,
}
```

### Phase 5: FPS Metrics
**New file:** `src/metrics.rs`

Real-time monitoring:
- `current_fps: AtomicU64` - actual frames/second
- `total_drops: AtomicU64` - cumulative dropped frames
- `drop_rate()` - percentage of dropped frames

Wire to `AppState` for potential UI/tray display.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Dropped frames cause playback issues | Only drop frames, never corrupt keyframes |
| FPS oscillation | 3-second cooldown, hysteresis thresholds |
| Extra thread overhead | Minimal - one writer thread per encoder |
| Memory for write queue | Bounded queue (16 frames max) |
